# Проект QdrantPayloadTest

Данный проект представляет собой пример реализации RAG (Retrieval-Augmented Generation) системы с использованием **Qdrant** в качестве векторной базы данных, **Ollama** для эмбеддингов и генерации ответов, а также модели **qwen2.5:14b** для анализа запросов и формирования ответов. Основная задача – индексация структурированных данных (сущности и связанные с ними даты) с последующим гибким поиском по датам и именам сущностей.

## 📁 Структура проекта

```
QdrantPayloadTest/
├── .venv/                     # виртуальное окружение Python
├── data/                       # каталог с данными
│   ├── entities/                # файлы сущностей (по одному на сущность)
│   │   ├── 1_name.txt
│   │   ├── 2_name.txt
│   │   ├── 3_name.txt
│   │   └── ...
│   ├── create_data.py           # скрипт для разбиения исходного файла на отдельные сущности
│   └── date_in_format_dd_mm.txt # исходный большой файл с датами
├── ollama_models/               # монтируемая папка для хранения моделей Ollama (используется в Docker)
├── qdrant_storage/              # папка для хранения данных Qdrant
├── heap.py                      # скрипт индексации (загрузка данных в Qdrant)
├── search.py                    # скрипт поиска с интеллектуальным парсингом запроса
├── docker-compose.yaml          # конфигурация Docker для запуска Qdrant и Ollama
└── README.md                    # этот файл
```

## 🧰 Требования

- **Python** 3.9 или выше
- **Docker** и **Docker Compose** (для запуска Qdrant и, опционально, Ollama)
- **Ollama** (рекомендуется устанавливать на хосте для избежания проблем с памятью, но можно и в Docker)
- Модели Ollama: `nomic-embed-text` (для эмбеддингов) и `qwen2.5:14b` (для генерации и парсинга). При использовании Docker модели скачиваются внутрь контейнера.

## 📊 Подготовка данных

Исходные данные должны быть представлены в виде файла с сущностями и датами. Пример формата (`date_in_format_dd_mm.txt`):

```
vehuiah (вехюиах)
Даты:
 21.03
 22.03
 23.03
 24.03
 25.03
 09.01
 21.03
 03.06
 17.08
 30.10

jeliel (иелиель)
Даты:
 26.03
 27.03
 28.03
 29.03
 30.03
 10.01
 22.03
 04.06
 18.08
 19.08
 31.10
...
```

Для удобства обработки такой файл разбивается на отдельные файлы по одной сущности. Для этого используется скрипт `create_data.py` (пример приведён ниже).

### Запуск разбиения

```bash
python data/create_data.py
```

Скрипт создаст в папке `data/entities/` файлы вида `vehuiah.txt`, `jeliel.txt` и т.д., где первая строка – имя сущности, остальные строки – даты.

Пример `create_data.py`:

```python
import os
import re

def split_entities(input_file: str, output_dir: str):
    os.makedirs(output_dir, exist_ok=True)
    with open(input_file, 'r', encoding='utf-8') as f:
        content = f.read()
    pattern = re.compile(r'^([a-z]+) \([^)]+\)\nДаты:\n((?:\s+\d{2}\.\d{2}\n?)+)', re.MULTILINE)
    for match in pattern.finditer(content):
        entity_name = match.group(1)
        dates_block = match.group(2).strip()
        dates = dates_block.split()
        output_content = f"{entity_name}\n" + "\n".join(dates)
        output_file = os.path.join(output_dir, f"{entity_name}.txt")
        with open(output_file, 'w', encoding='utf-8') as out_f:
            out_f.write(output_content)
        print(f"Создан файл: {output_file}")

if __name__ == "__main__":
    input_file = "data/date_in_format_dd_mm.txt"
    output_dir = "data/entities"
    split_entities(input_file, output_dir)
```

## 📥 Индексация данных в Qdrant

Скрипт `heap.py` выполняет следующие действия:

- Читает все файлы из папки `data/entities`.
- Для каждого файла извлекает имя сущности и список дат.
- Формирует текст для эмбеддинга (объединяет имя и даты).
- Получает эмбеддинг через Ollama (модель `nomic-embed-text`).
- Создаёт точку в коллекции Qdrant с payload, содержащим `entity_name`, `dates` и другую метаданные.
- Если коллекция ещё не существует, она создаётся с индексами на поля `dates` и `entity_name`.

### Запуск индексации

Перед запуском убедитесь, что коллекция пуста (или удалите старые данные). В скрипте предусмотрена очистка:

```bash
python heap.py
```

В консоли отобразятся сообщения о ходе обработки и загрузке точек.

## 🔍 Поиск

Скрипт `search.py` реализует интеллектуальный поиск:

1. **Парсинг запроса** с помощью модели `qwen2.5:14b`. Из запроса извлекаются:
   - даты (в формате ДД.ММ)
   - имена сущностей
   - свободный текст для семантического поиска

2. **Построение фильтра** для Qdrant на основе извлечённых дат и имён (используется логика AND между группами, OR внутри группы).

3. **Семантический поиск** – эмбеддинг вычисляется для свободного текста (или исходного запроса, если свободный текст пуст).

4. **Генерация ответа** – найденные сущности и их даты передаются в модель вместе с исходным вопросом для формирования ответа.

### Пример запуска

```bash
python search.py
```

Вы увидите распознанные компоненты запроса и сгенерированный ответ.

Пример запроса:  
`Каким именам соответствует дата 14.03?`  
Ответ перечислит все сущности, у которых в списке дат есть `14.03`.

## 🐳 Запуск с Docker (альтернативный вариант)

Если вы предпочитаете запускать все сервисы в Docker, используйте предоставленный `docker-compose.yaml`. Пример файла:

```yaml
version: '3.8'
services:
  qdrant:
    image: qdrant/qdrant:latest
    container_name: qdrant
    ports:
      - "6333:6333"
      - "6334:6334"
    volumes:
      - ./qdrant_storage:/qdrant/storage
    restart: unless-stopped

  ollama:
    image: ollama/ollama:latest
    container_name: ollama
    ports:
      - "11434:11434"
    volumes:
      - ./ollama_models:/root/.ollama
    restart: unless-stopped
    command: serve
```

**Важно:** Для корректной работы модели `qwen2.5:14b` в контейнере необходимо выделить достаточно оперативной памяти. В Docker Desktop для macOS откройте настройки → Resources → Advanced и увеличьте Memory до 16–20 ГБ. Также можно добавить в `docker-compose.yaml` ограничения:

```yaml
services:
  ollama:
    ...
    deploy:
      resources:
        reservations:
          memory: 16G
```

Однако ограничения `deploy` работают только в swarm mode, поэтому проще настроить глобальные лимиты Docker.

## ⚠️ Возможные проблемы и решения

### 1. Ошибка 404 при запросе к Ollama
Убедитесь, что Ollama запущена и доступна на `localhost:11434`. Проверьте командой `curl http://localhost:11434/api/tags`. Если Ollama в Docker, убедитесь, что контейнер работает.

### 2. Ошибка 500 при генерации ответа (not enough memory)
Модель `qwen2.5:14b` требует около 8–9 ГБ свободной оперативной памяти. При использовании Docker на macOS лимиты Docker могут быть меньше. Решения:
- Увеличить лимит памяти в настройках Docker Desktop.
- Запустить Ollama на хосте (вне Docker).
- Использовать меньшую модель, например `qwen2.5:7b` (измените имя модели в `search.py`).

### 3. Ошибка парсинга JSON от LLM
Иногда модель может вернуть ответ, не являющийся валидным JSON. В скрипте предусмотрен fallback на пустую структуру. Если это происходит часто, попробуйте уменьшить `temperature` до 0 и уточнить промпт.

### 4. Долгая загрузка моделей
При первом запуске модели скачиваются (размер `qwen2.5:14b` ~9 ГБ). Подождите завершения.

## 📝 Примечания

- Все скрипты используют URL `http://localhost:11434` для обращений к Ollama. Если Ollama запущена в Docker на той же машине, этот URL работает.
- В скрипте `heap.py` используется простая регулярка для извлечения дат. Для реальных проектов рекомендуется более мощный парсер (например, Natasha).
- Проект является демонстрационным и может быть адаптирован под любые структурированные данные с метаданными.
